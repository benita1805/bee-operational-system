BEE OPERATIONAL SYSTEM (BUZZ-OFF) - FILE ROLES & LOGIC

This document describes the specific role and business logic 
implemented in major files of the Bee Operational System.

1. BACKEND ARCHITECTURE (Node.js/TypeScript)
------------------------------------------------------------------------

/src/services
-------------
- googleSheets.service.ts
  * ROLE: Core Data Access Object (DAO).
  * LOGIC: Manages HTTP communication with a Google Apps Script bridge. 
    It translates generic CRUD requests (create, find, list, update) 
    into Google Sheets API calls, treating spreadsheet tabs as tables.

- hive.service.ts
  * ROLE: Hive Domain Logic.
  * LOGIC: Handles ownership verification (ensuring users only see 
    their own hives). Implements the synchronization algorithm:
    compares 'updated_at' timestamps between client and server to 
    resolve conflicts using a last-write-wins strategy.

- auth.service.ts
  * ROLE: Authentication Handler.
  * LOGIC: Manages the lifecycle of One-Time Passwords (OTP). It 
    generates 6-digit codes, stores them temporarily, and issues 
    JWT (JSON Web Tokens) upon successful verification.

- farmer.service.ts
  * ROLE: Farmer Management Logic.
  * LOGIC: Handles registration and lookup for farmer entities. 
    Links farmers to geographic coordinates for mapping.

/src/controllers
----------------
- auth.controller.ts, hive.controller.ts, farmer.controller.ts
  * ROLE: Request Orchestration.
  * LOGIC: Extracts data from the incoming HTTP request (params, body, 
    user context), calls the appropriate service methods, and 
    standardizes the JSON response format.

/src/middlewares
----------------
- auth.middleware.ts
  * ROLE: Security Guard.
  * LOGIC: Decodes the JWT from the Authorization header. If valid, 
    it attaches the user ID to the request object; otherwise, 
    it blocks the request with a 401 Unauthorized error.

- errorHandler.ts
  * ROLE: Centralized Error Management.
  * LOGIC: Catches all unhandled exceptions, logs them to the console, 
    and returns a clean, user-friendly error response to the client.

2. FRONTEND ARCHITECTURE (React Native/Expo)
------------------------------------------------------------------------

/src/screens
------------
- DashboardScreen.js
  * ROLE: Executive Summary.
  * LOGIC: Aggregates data from hives and farmers to show real-time 
    metrics (e.g., hives due for harvest in the next 7 days). 
    Provides quick-action navigation via Expo Router.

- MapScreen.js
  * ROLE: Geographic Interface.
  * LOGIC: Uses MapView to plot farmer locations. Implements 
    interactive markers that reveal farmer details and crop info.

- /Hives/MyHivesScreen.js & HiveFormScreen.js
  * ROLE: Hive Lifecycle Management.
  * LOGIC: Manage local state for hive records. The Form screen 
    handles complex validation (required fields) before sending 
    data to the backend.

/src/services
-------------
- apiClient.ts
  * ROLE: Network Wrapper.
  * LOGIC: A centralized fetch utility that automatically injects the 
    user's authentication token into every outgoing request.

- hiveCloud.ts & farmersApi.js
  * ROLE: Service Adapters.
  * LOGIC: Encapsulate the API endpoints. They handle the async/await 
    logic and provide error boundaries to prevent UI crashes if 
    network requests fail.

- aiEngine.js
  * ROLE: Intellectual Layer.
  * LOGIC: Analyzes hive logs and health data on the client side 
    to generate "AI Risk Scores" and management suggestions.

/src/utils
----------
- location.js: Handles user permissions and GPS coordinate fetching.
- distance.js: Calculates the distance between the user and farmers.
- date.ts: Manages harvest date calculations and ISO string formatting.
